# 网页数据获取 MCP 服务 - 需求文档（v1）

> 状态：草案 | 日期：2026-07-06 | 阶段：第一阶段（MVP）

---

## 一、项目概述

### 1.1 一句话定位

一个**自托管的远程 MCP 服务**，部署在个人服务器上，通过 HTTP 提供「联网搜索 + 网页获取」能力给 AI 编程工具（Claude Code / Cursor 等），并配套一个网页端管理后台用于 Key 管理和用量统计。

### 1.2 背景与目标

- **背景**：AI 编程工具需要联网搜索能力，但内置工具是黑盒、不可定制。已有闲置服务器可部署服务。
- **目标**：
  - 自己的 AI 工具能通过一个 URL + Key 调用搜索/获取能力
  - 后期可分享给几个信任的朋友共同使用
  - 全程自己掌控（搜索引擎、数据、用量）
- **非目标**：不做 ToC 商业化、不做客户端 App。

### 1.3 与原设计文档的关系

原 `web-fetch-mcp-server-design.md` 基于「本地 stdio MCP」设计。本需求将形态调整为「远程 HTTP MCP + 管理后台」，原文档中的**搜索逻辑、HTML→Markdown、SSRF 防护**等仍可作为实现参考。

## 二、用户角色与使用场景

| 角色 | 是谁 | 如何使用 |
|------|------|---------|
| **管理员** | 你自己 | 登录网页后台，管理 Key、查看统计 |
| **调用方** | 你自己 / 信任的朋友 | 在 AI 工具中配置 MCP URL + Key 调用 |

**典型场景：**
1. 你在后台创建一个 Key（标注用途，如"我的 Claude Code"）
2. 把 MCP 服务 URL + Key 配置到 AI 工具
3. AI 工具自主调用搜索/获取能力
4. 后台实时看到该 Key 的请求次数

## 三、功能需求

### 3.1 MCP 工具能力（给 AI 工具调用）

通过 HTTP/SSE 暴露 3 个工具，所有调用必须携带有效 API Key：

| 工具 | 输入 | 输出 | 说明 |
|------|------|------|------|
| **`web_search`** | 关键词、结果数量、时间范围 | 结构化结果列表（标题、URL、摘要） | 基于 SearXNG |
| **`web_fetch`** | URL、是否浏览器渲染 | Markdown 格式正文 | 含 SSRF 防护、内容截断 |
| **`web_search_and_fetch`** | 关键词、获取数量 | 搜索结果 + 批量页面正文 | 搜索+获取一次完成 |

> 调用方只需配置 URL + Key，AI 工具自动发现并决定何时调用哪个工具。

### 3.2 管理后台功能（网页端，Vue SPA）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **管理员登录** | 单一管理员密码登录（仅一个账号） | P0 |
| **请求总览** | 查看服务总请求次数（全局统计，可按工具/时间维度） | P0 |
| **Key 管理 - 新增** | 创建新 Key，可标注名称（如「老王」「我的 Claude Code」） | P0 |
| **Key 管理 - 查看** | 列出所有 Key 及其状态、创建时间 | P0 |
| **Key 管理 - 启停/删除** | 禁用或删除某个 Key | P0 |
| **Key 用量统计** | 查看每个 Key 的请求次数（可按工具/时间维度） | P0 |
| Key 用量趋势图 | 按天/小时的请求趋势可视化（图表） | P1（可后置） |

> **Key 语义**：一个 Key 代表一个「使用方」。例如你自己可能有多个 Key：「我的 Claude Code」「我的 Cursor」「老王」。每个可单独启停、独立统计。

### 3.3 鉴权机制

| 场景 | 鉴权方式 |
|------|---------|
| AI 工具调用 MCP（远程） | **API Key**（每个调用方一个，请求头携带） |
| 管理员访问后台 | **管理员密码**（登录后用 Session/JWT） |

## 四、非功能需求

| 维度 | 要求 |
|------|------|
| **安全** | URL 抓取必须做 SSRF 防护（防内网/元数据端点）；服务对外走 HTTPS；Key 以哈希存储 |
| **可用性** | SearXNG 容器崩溃可自动重启；服务异常不静默失败 |
| **性能** | 单次搜索 < 3s；网页获取 < 5s（不含浏览器渲染） |
| **可维护性** | 配置通过环境变量管理；日志可查 |
| **资源** | 服务器资源有限，Playwright（浏览器）默认关闭，按需启用 |

## 五、技术架构（概要）

### 5.1 部署形态

```
[AI 工具] --HTTPS+Key--> [你的服务器]
                            ├── MCP 服务（HTTP/SSE）---> SearXNG / 目标网页
                            └── Web 管理后台 ---> 数据库
```

### 5.2 组件划分

| 组件 | 职责 | 说明 |
|------|------|------|
| **MCP 服务** | 暴露 3 个工具给 AI，校验 Key、记录用量 | 远程 HTTP MCP |
| **Web 后台（后端 API）** | 登录、Key CRUD、统计查询 | 提供 REST API |
| **Web 后台（前端）** | 管理界面网页 | 浏览器访问 |
| **搜索后端** | SearXNG 容器 | 外部依赖，Docker 部署 |
| **数据库** | 存储 Key、用量日志 | 建议 SQLite（见待确认项） |

### 5.3 技术栈（已确认）

| 层 | 选型 | 理由 |
|----|------|------|
| 语言/运行时 | TypeScript + Node.js | MCP SDK 一等支持 |
| MCP SDK | `@modelcontextprotocol/sdk` | 官方 |
| 搜索后端 | SearXNG（独立容器） | 已确认，与呆呆项目完全隔离 |
| 数据库 | **MySQL 8.0（独立容器）** | 与呆呆完全物理隔离，独立网络、独立实例 |
| ORM | Prisma | TS 体验最好，迁移管理清晰 |
| 后台后端 | Node.js + Fastify | 提供 REST API（登录/Key/统计） |
| 后台前端 | **Vue 3 + Vite** SPA | 配 Element Plus / Naive UI |
| 反向代理 | **复用现有 honlnk-gateway (Nginx)** | 不新建 Nginx，加子域名配置接入 |

### 5.4 服务部署形态（融入现有 gateway 体系）

**核心原则：与呆呆项目完全隔离，但复用 honlnk-gateway 这个统一入口。**

```
火山云服务器 (Ubuntu 24.04, 115.190.219.215)
│
├── honlnk-gateway (现有 Nginx, 容器, 80/443)   ← 复用，仅加配置
│     ├── daidai.honlnk.top   → 现有呆呆业务（不动）
│     ├── piclist.honlnk.top  → 现有图床（不动）
│     ├── oss.honlnk.top      → 现有对象存储（不动）
│     └── search.honlnk.top 🆕 → 新服务
│           ├── /        → Vue 后台前端（静态文件，由 gateway 托管）
│           ├── /api/*   → web-fetch-app 后台 REST API
│           └── /mcp     → web-fetch-app MCP 服务 (HTTP/SSE)
│
├── 【独立网络 web-fetch-net 🆕】（与 dai-dai-net 隔离）
│   ├── web-fetch-app 容器 🆕 (Node.js：MCP + 后台 API)
│   ├── web-fetch-mysql 容器 🆕 (MySQL 8.0，独立实例)
│   └── web-fetch-searxng 容器 🆕 (SearXNG)
│
└── （呆呆项目的容器和网络完全不动）
```

**网络隔离方案：**
- 新建独立 Docker 网络 `web-fetch-net`
- `web-fetch-app` 同时接入 `web-fetch-net` 和 `honlnk-gateway-net`（前者连 DB/搜索，后者让 gateway 能反代到它）
- 呆呆项目的 `dai-dai-net` 完全不碰

**资源预算（服务器剩余 ~1.6G 内存）：**

| 新增容器 | 预估内存 |
|---------|---------|
| web-fetch-app (Node.js) | ~150MB |
| web-fetch-mysql (MySQL 8.0) | ~400MB |
| web-fetch-searxng | ~100MB |
| **合计** | **~650MB**，剩余约 950MB，够用 |

### 5.5 子域名与证书

- **子域名**：`search.honlnk.top`（DNS 需 A 记录指向 `115.190.219.215`）
- **证书**：复用 `/home/honlnk/honlnk-gateway/certs/` 下现有的泛域名证书（`*.honlnk.top`）
- **Nginx 配置**：在 `/home/honlnk/honlnk-gateway/conf.d/` 新增 `search.honlnk.top.conf`

## 六、第一阶段范围边界

### ✅ 第一阶段包含
- 3 个 MCP 工具（搜索 / 获取 / 搜索并获取）
- API Key 鉴权
- 管理后台：登录、总请求数、Key 增删查、Key 用量
- SearXNG 部署
- HTTPS + Nginx 反代
- 基础 SSRF 防护

### ❌ 第一阶段不包含（后续扩展）
- Playwright 浏览器渲染（先靠 HTTP，覆盖 90% 场景）
- 缓存（LRU）
- 域名白/黑名单
- 多管理员账号
- 用量趋势图表、告警
- 计费/配额限制
- 缓存持久化、RAG 集成

### 8.2 已有服务（不动它）

服务器上已经在跑一套呆呆业务 + 工具：

| 容器 | 用途 | 端口 |
|------|------|------|
| `honlnk-gateway` (nginx) | **统一入口网关** | 80, 443 |
| `dai-dai-gateway/backend/admin-web` | 呆呆业务 | 8080→80, 8443→443 |
| `dai-dai-mqtt` | MQTT | 1883, 8883 |
| `mysql-dev` | 呆呆用的 MySQL 8.0 | 3001→3306（**新项目不复用**） |
| `redis-dev` | Redis 7 | 3101→6379 |

### 8.3 关键设施复用

| 设施 | 现状 | 复用方式 |
|------|------|---------|
| **Nginx 网关** | `honlnk-gateway` 容器，已在 `honlnk-gateway-net` 网络 | 新 app 加入该网络，加一个 server 配置 |
| **SSL 证书** | `/home/honlnk/honlnk-gateway/certs/{fullchain,privkey}.pem`（泛域名 `*.honlnk.top`） | 直接挂载/引用 |
| **Nginx 配置目录** | `/home/honlnk/honlnk-gateway/conf.d/`（挂载到容器 `/etc/nginx/conf.d`，只读） | 新增 `search.honlnk.top.conf` |
| **域名规律** | `*.honlnk.top` / `*.daidai.honlnk.top` | 新增 `search.honlnk.top` 子域名 |

### 8.4 部署约束

- **内存紧**：剩余 ~1.6G，新项目三个容器合计约 650MB，**Playwright 严禁默认开启**
- **网络隔离**：新项目走独立网络 `web-fetch-net`，**不接入 `dai-dai-net`**
- **MySQL 独立**：新起 `web-fetch-mysql` 容器，与呆呆 `mysql-dev` 物理隔离
- **不动现有服务**：部署和调试过程中不重启/不修改呆呆相关任何容器与配置
